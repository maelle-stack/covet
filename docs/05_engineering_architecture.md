Covet must be built as a production-grade iOS-first application using React Native, Expo, TypeScript, Supabase, Postgres, and a backend service layer. The architecture should prioritize correctness, trust, maintainability, security, and fast iteration. This is not a prototype architecture. The app will handle sensitive financial, calendar, behavioral, and identity data, so the codebase must be structured as if it will support millions of users and survive years of product development. Claude/Codex should not build temporary flows, duplicated calculations, mock-only service logic, or UI-owned business rules. All financial reasoning must live in shared engine services, not inside screens.
The recommended stack is React Native with Expo for the iOS-first mobile app, TypeScript across the frontend and backend, Supabase Auth and Postgres for authentication and persistence, a backend service layer for financial logic and integrations, Plaid for bank data, Google Calendar OAuth through backend services, Apple Calendar permissions through the iOS client where appropriate, TanStack Query for server state, Zustand for lightweight local UI/session state, an AI service abstraction for model-swappable intelligence features, backend Safe to Spend calculation as the source of truth, cached SafeToSpendSnapshot objects on-device for fast Home loading, and Expo/APNs push notification infrastructure depending on the final build path. The app should be designed for iOS first. Web can come later as a secondary surface and should not compromise the mobile architecture.
The frontend should be organized by product surface and shared systems. A suggested structure is src/app for navigation and app shell, src/screens for Home, Activity, Upcoming, Purchase Check, Onboarding, Settings, and Auth, src/componentsfor reusable UI, src/design for tokens and motion constants, src/features for domain-specific modules, src/services for API clients and local integrations, src/state for Zustand stores, src/hooks for reusable hooks, src/types for shared TypeScript types, and src/utils for pure helpers. Product logic should not be embedded in screen files. Screens should compose data and components; they should not calculate Safe to Spend, classify debt pressure, decide notification severity, or infer recurring habits.
The backend should be organized as domain services. A suggested structure is services/auth, services/plaid, services/calendar, services/financial-engine, services/notification-engine, services/context-engine, services/insights-agent, services/purchase-check, services/vaults, services/sync, and services/privacy. Each service should have clear responsibilities and typed inputs/outputs. The financial engine should be pure and testable wherever possible. Integration services can handle external APIs, tokens, sync jobs, and webhooks. No external integration should leak directly into UI code.
The database should use Postgres as the source of truth. Supabase is acceptable for authentication, database access, row-level security, and realtime updates where useful, but sensitive server-side operations should still go through backend functions or service routes. The client should never hold Plaid secrets, calendar OAuth secrets, notification signing credentials, or AI provider secrets. Any operation involving external API secrets, financial calculation, notification dispatch, or model calls must go through backend-controlled services.
The core production data models are User, UserSettings, AuthIdentity, BankConnection, Account, Transaction, CalendarConnection, CalendarEvent, Commitment, RecurringItem, Pattern, ArchetypeResult, SafeToSpendSnapshot, PurchaseCheck, Vault, Notification, Insight, SyncJob, and AuditEvent. These should be modeled as durable entities, not ad hoc JSON blobs scattered across screens. JSON metadata can exist for extensibility, but critical fields must be typed and queryable. Anything that affects Safe to Spend, notifications, commitments, or user trust must be stored in structured fields.
The User model should store stable identity fields and product-level profile data. It should include id, createdAt, updatedAt, firstName, timezone, locale, onboardingStatus, primaryArchetype, secondaryArchetype, activeGoal, strictnessLevel, and accountStatus. Sensitive authentication fields should be handled by the auth provider and not duplicated unnecessarily. The UserSettings model should include notification preferences, quiet hours, privacy preview level, daily pacing enabled/disabled, sale alerts enabled/disabled, vault notifications enabled/disabled, biometric lock preference, calendar preferences, and data/privacy preferences.
The BankConnection model should store the connection state for Plaid-linked institutions. It should include id, userId, provider, providerItemId, institutionName, status, lastSuccessfulSyncAt, lastWebhookAt, requiresReauth, errorCode, and createdAt. Tokens must be stored securely on the backend and never exposed to the client. The Account model should include id, userId, bankConnectionId, providerAccountId, name, officialName, type, subtype, maskLast4, currentBalance, availableBalance, isoCurrencyCode, lastSyncedAt, and status. Checking accounts are the primary source for Safe to Spend. Savings should not be included unless a future product decision explicitly allows it.
The Transaction model should include id, userId, accountId, providerTransactionId, amount, merchantName, originalDescription, category, subcategory, date, authorizedDate, pending, type, isoCurrencyCode, paymentChannel, confidence, isTransfer, excludedFromIncome, excludedFromSpending, recurringCandidateId, and metadata. Pending transactions must reduce Safe to Spend immediately. Transfers should be excluded from income and spending by default, but unclear transfers can create review actions. Transaction deduplication is required because Plaid and pending-to-posted transitions can otherwise cause false spending.
The CalendarConnection model should include id, userId, provider, status, lastSuccessfulSyncAt, requiresReauth, and selectedCalendars. CalendarEvent should include id, userId, calendarConnectionId, providerEventId, title, startAt, endAt, timezone, location, isVirtual, virtualProvider, travelDistanceEstimate, travelMode, notesAvailable, spendCandidate, confidence, linkedCommitmentId, and metadata. Virtual events such as Zoom meetings should not become financial commitments unless user context or observed spending proves otherwise. In-person events, travel, birthdays, concerts, exams, trips, and social plans can become candidate commitments requiring approve/deny controls.
The Commitment model should represent anything Covet is actively protecting or considering. It should include id, userId, source, title, amount, estimatedAmount, confirmedAmount, dueAt, commitmentType, hardness, status, protectedAmount, protectionStartAt, confidence, userConfirmed, userDenied, linkedCalendarEventId, linkedTransactionId, linkedRecurringItemId, and metadata. Commitment hardness should support hard, semi-hard, and soft. Hard commitments include rent, essential bills, insurance, required minimum debt payments, and non-negotiable obligations. Semi-hard commitments include confirmed events, travel, ID replacement, application fees, and important user-entered expenses. Soft commitments include brunch, Pilates, coffee, beauty, shopping, and similar lifestyle habits.
The RecurringItem model should represent bills, subscriptions, and recurring habits together. It should include id, userId, title, merchantName, amountEstimate, cadence, nextExpectedAt, recurringType, hardness, confidence, status, linkedPatternId, lastSeenAt, userConfirmed, userPaused, and metadata. Recurring habits should be allowed to affect Safe to Spend as soft commitments but should be reducible when money is tight. Recurring bills and subscriptions should usually be protected more strongly.
The Pattern model should store detected behavioral and financial patterns. It should include id, userId, type, description, confidence, evidenceSummary, status, firstDetectedAt, lastDetectedAt, confirmedAt, deniedAt, and relatedEntityIds. Patterns should not automatically become product truth. They should influence candidate commitments, recurring items, insights, and actions only according to confidence and user confirmation rules. Pattern detection should be testable and should never rely on vague LLM interpretation alone when deterministic transaction evidence is available.
The ArchetypeResult model should store onboarding quiz results. It should include userId, primaryArchetype, secondaryArchetype, quizVersion, answers, createdAt, and updatedAt. The engine should initially use the primary archetype for behavior buffer configuration. Secondary archetype should appear in user-facing identity and may influence future model refinement after enough behavior is observed. Archetype should be allowed to evolve over time based on user behavior, but changes should not happen silently in a way that confuses the user.
The SafeToSpendSnapshot model is one of the most important objects in Covet. It should be generated by the backend financial engine and consumed by Home, Purchase Check, Notifications, Activity, and Upcoming. It should include id, userId, amount, payCycleStart, payCycleEnd, daysUntilNextIncome, dailyPace, internalProjectedPace, status, confidenceScore, externalConfidenceLabel, protectedHardCommitments, protectedSoftCommitments, debtPressureLevel, obligationPressureLevel, emergencyFloorApplied, behaviorBufferApplied, majorChangeFlags, explanationSummary, lastCalculatedAt, staleAfter, and inputsHash. The client must never recalculate Safe to Spend independently. If the snapshot is stale, the UI should show the last known number with an appropriate confidence/staleness label.
The PurchaseCheck model should include id, userId, inputType, rawInput, parsedItemName, parsedMerchant, parsedPrice, parsedUrl, screenshotAssetId, decision, decisionReason, safeToSpendBefore, safeToSpendAfterHypothetical, relatedVaultId, statusAtDecision, createdAt, and followUpAt. Purchase Check should support text in v1 and should be architected for screenshots and links. If the user asks about an item and Covet says wait, the system should be able to notify later when the item becomes safe.
The Vault model should include id, userId, title, targetAmount, currentProtectedAmount, desiredByDate, status, activelyProtected, source, merchant, url, imageAssetId, affordabilityDate, lastRecalculatedAt, and notificationPreferences. Vaults should reduce Safe to Spend only when actively protected. Passive desires should not behave like mandatory obligations.
The Notification model should include id, userId, candidateEventId, triggerType, severity, title, body, privacyBody, createdAt, scheduledFor, sentAt, quietHoursApplied, batchedEventIds, suppressedReason, statusBefore, statusAfter, safeToSpendBefore, safeToSpendAfter, relatedCommitmentId, relatedVaultId, relatedPatternId, relatedPurchaseCheckId, actionType, and userResponse. Notifications should be generated by the Notification Engine, not screens. The UI can display notification history only if useful, but Activity should not become a raw notification log.
The Insight model should include id, userId, insightType, title, body, evidenceSummary, confidence, generatedAt, expiresAt, status, relatedTransactions, relatedPatterns, relatedCommitments, and userFeedback. Insights should not generate until the user has at least 25 transactions inside Covet. Insights should be produced by a separate Insights Agent and grounded in user-specific data. The app should not cite books or behavioral frameworks to the user. The output should feel like Covet noticed something specific.
The SyncJob model should track Plaid syncs, calendar syncs, notification checks, financial recalculations, insight generation, and cleanup jobs. It should include id, userId, jobType, status, startedAt, completedAt, errorCode, errorMessage, retryCount, and metadata. The AuditEvent model should track important security, data, and financial-engine events for debugging and trust. It should include id, userId, eventType, entityType, entityId, createdAt, and metadata. Audit logs should not expose sensitive values unnecessarily.
State management should separate server state from local UI state. TanStack Query should manage server-derived data such as SafeToSpendSnapshot, Activity, Upcoming, Transactions, Notifications, Settings, and PurchaseCheck results. Zustand should manage lightweight client state such as active tab, onboarding step, temporary form inputs, local UI preferences, and unsaved chat input. Persistent local storage should be used carefully and should not store raw financial data beyond approved cached snapshots and necessary app state. Sensitive local data should be minimized and protected.
The Home screen must load from the most recent cached SafeToSpendSnapshot immediately, then refresh from the backend. This ensures the app feels fast even when network or sync is slow. If the cached snapshot is stale, disconnected, or low confidence, the UI should show that discreetly. The user should never see an empty Home simply because the network is slow if a recent snapshot exists. Offline mode should show last known Safe to Spend with a clear stale-data label and should disable or qualify actions that require fresh financial data.
Plaid integration must run through backend services only. The client can initiate Plaid Link and receive a public token, but the exchange for access tokens and all item/account/transaction sync logic must happen on the backend. Plaid webhooks should trigger sync jobs, transaction reconciliation, Safe to Spend recalculation, and notification candidate generation. The backend must handle pending-to-posted transaction updates, duplicate transactions, disconnected items, reauth requirements, institution errors, stale data, and partial sync failures.
Calendar integration should support Google Calendar through backend OAuth and Apple Calendar through local iOS permission flow where appropriate. Calendar data should be normalized into CalendarEvent records. Calendar sync should classify events as virtual, in-person, travel-related, academic, social, birthday, appointment, or unknown when possible. The system should not create financial commitments from every calendar event. It should create candidate commitments only when there is plausible financial impact and enough confidence or user confirmation.
The Safe to Spend engine should run on the backend as the source of truth. It should be invoked after transaction sync, bank balance updates, confirmed or denied commitments, recurring item changes, purchase checks, calendar updates, income cadence changes, debt updates, vault activation, and major pattern updates. It should also run on a scheduled cadence, including at the start of a new day, because days until payday and due dates change even without new transactions. The engine should produce a new SafeToSpendSnapshot only when inputs materially change or when a scheduled recalculation changes time-based fields.
The Notification Engine should run on backend-generated candidate events. It should apply severity, materiality thresholds, quiet hours, privacy settings, batching, suppression, and delivery rules. Push notifications should use Expo notifications or APNs depending on the final build path. The architecture should not assume push delivery always succeeds. Notification records must track scheduled, sent, suppressed, opened, and acted-on states where possible. The engine should also support notification privacy levels: Full detail, Discreet, and Hidden.
The AI service layer should be abstracted behind a model-agnostic interface. Purchase parsing, screenshot/link understanding, behavioral insights, natural-language explanations, and copy generation should call an internal AI service interface rather than directly depending on one model provider. The interface should support provider, model, promptVersion, input, output, confidence, latency, tokenCost, and safety/error metadata. Critical financial calculations should not depend on the LLM. The LLM may explain, parse, summarize, or generate insights, but deterministic engine services must own final money decisions.
API contracts should be typed and stable. Suggested endpoint groups include /auth, /user, /onboarding, /bank-connections, /calendar-connections, /transactions, /commitments, /recurring, /patterns, /safe-to-spend, /purchase-checks, /vaults, /notifications, /insights, /settings, and /privacy. Each endpoint should have typed request and response schemas. The frontend should use generated or shared TypeScript types wherever possible. Avoid loosely typed API responses because financial products degrade quickly when frontend and backend interpretations diverge.
The /safe-to-spend/current endpoint should return the latest SafeToSpendSnapshot for the user. The /safe-to-spend/recalculateendpoint should trigger or request recalculation only when appropriate and should be protected against spam. The /purchase-checks/create endpoint should accept text, link, or screenshot metadata and return a PurchaseCheck decision. The /commitments/:id/confirm, /commitments/:id/deny, and /commitments/:id/edit endpoints should update commitment status and trigger recalculation. The /patterns/:id/confirm and /patterns/:id/deny endpoints should update pattern state and any linked recurring items. The /notifications/preferences endpoint should manage privacy level, quiet hours, daily pacing, sale alerts, and vault alerts.
Testing is mandatory. The Financial Engine must have unit tests for Safe to Spend formula, emergency floor, daily pace, internal projected pace, behavior buffer, commitment protection, debt pressure tiers, income cadence, irregular income, obligation pressure, pending transactions, transfers, vault activation, status mapping, and confidence scoring. The Notification Engine must have tests for trigger types, materiality thresholds, quiet hours, batching, suppression, privacy previews, connection disconnects, purchase check follow-ups, commitment risk, and sale alert toggles. Tests should include realistic user scenarios, not only isolated functions.
Scenario tests should cover representative Covet users. Examples include a high school student with $500 income and almost no obligations, a student with $500 income and $400 mandatory obligations, an early-career worker paid biweekly with rent and brunch habits, a freelancer with irregular income, a user with high credit utilization, a user with pending transactions, a user with a confirmed birthday dinner, a user in exam week, and a user asking whether they can buy a $180 jacket. These tests should assert Safe to Spend, status, confidence, notification candidates, and explanation summaries.
Security and privacy must be treated as product requirements, not a later compliance pass. Covet should never sell user financial or calendar data. User data should be used only to deliver the requested product experience: Safe to Spend, commitment protection, notifications, insights, purchase checks, and related money management features. The app should support account deletion, data deletion, and export where required. Sensitive tokens must be encrypted or stored using secure provider facilities. Row-level security and backend authorization checks must prevent cross-user data access. Logs should avoid storing raw sensitive financial details unless necessary and protected.
The app must support privacy-safe notifications by default. Lock screen previews should default to discreet. Exact Safe to Spend amounts, debt pressure, embarrassing repetitive behaviors, and sensitive merchant/category data should not appear on the lock screen unless the user chooses Full detail. The notification object should store both full body and privacyBody so delivery can respect user settings.
Error handling should be calm and specific. If bank data is disconnected, tell the user the connection needs refreshing and Safe to Spend may be less accurate. If calendar data is disconnected, tell the user Upcoming may miss plans. If Plaid sync fails, retry and expose only useful messages. If Safe to Spend cannot be calculated, do not invent a number. Show the last known snapshot if available, mark it as stale, and guide the user to fix the cause. Trust is more important than always showing a polished state.
The production build should include observability for sync failures, engine errors, notification suppression, push delivery, Plaid webhook processing, calendar sync, AI service latency, and Safe to Spend recalculation time. Internal logs and analytics should help the team understand whether Covet is accurate, quiet, and useful. Analytics should not encourage building engagement loops that conflict with the product goal. Measure trust, action, and reduced anxiety rather than raw screen time alone.
Part 5 acceptance criteria are strict. The app must be iOS-first and built with React Native, Expo, and TypeScript. The backend must own financial calculations, Plaid secrets, calendar OAuth secrets, AI provider secrets, notification generation, and Safe to Spend snapshots. The UI must never independently calculate Safe to Spend. Postgres must store structured models for users, accounts, transactions, commitments, recurring items, patterns, vaults, purchase checks, notifications, insights, calendar events, settings, and snapshots. Plaid must sync through backend services. Calendar data must be normalized before becoming candidate commitments. TanStack Query must handle server state, and Zustand may handle lightweight UI state. Offline Home must show the last known SafeToSpendSnapshot with staleness. Financial and notification engines must be covered by unit and scenario tests. The architecture must be modular enough that future systems like subscription cancellation, auto-save, auto-pay, card products, concierge, and investing can be added without rewriting the core.
