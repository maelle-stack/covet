The Financial Engine is the core decision system inside Covet. It is responsible for turning bank data, income patterns, spending behavior, debt, recurring obligations, calendar commitments, user goals, and archetype data into a simple answer: what the user can safely spend during the current pay cycle. The engine must be deterministic, testable, explainable at a high level, and separate from the interface. The UI may feel calm and minimal, but the engine underneath must be rigorous. Covet should never guess casually with money. When the engine is unsure, it should lower confidence, become more protective, and communicate that it is still learning.
Safe to Spend is the primary output of the Financial Engine. It represents the amount the user can safely spend during the current pay cycle after protecting what matters. It is not the user’s checking balance. It is not available credit. It is not a category budget. It is not net worth. Safe to Spend is a judgment number based on usable cash, confirmed and predicted obligations, behavior risk, debt pressure, emergency protection, and timing. The user sees this as a dollar amount on Home, supported by a daily pace and a status such as YOU’RE GOOD, TAKE IT EASY, WAIT UNTIL PAYDAY, or LET’S NOT….
The v1 Safe to Spend formula should begin with checking account cash only. Savings should not be included unless a future version explicitly allows the user to mark a savings source as usable. Credit card available balance should never increase Safe to Spend. Covet should not make users feel richer because they have more credit available. Credit cards may reduce Safe to Spend through debt pressure, required minimum payments, and utilization risk. If Covet observes that a user consistently pays their card off in full or near-full, on time, and keeps utilization healthy, the engine may apply a conservative trusted payoff adjustment by reducing debt pressure. This is not a credit boost. It is a small reduction in penalty because the user has demonstrated responsible payoff behavior.
The high-level formula is: Safe to Spend = usable checking cash minus emergency floor minus protected commitments minus recurring bills and habits minus behavior buffer minus debt pressure adjustment plus eligible confirmed incoming income within the current pay cycle. This formula should be implemented as configurable engine logic rather than scattered UI calculations. All Safe to Spend outputs should be rounded down to avoid false precision and to keep the app conservative. If the engine produces a negative number, the UI should not show a fake positive amount. It should show the appropriate protective status and a clear explanation that the user is tight until income arrives or until a commitment is reduced.
The emergency floor starts at 10% of checking. This floor exists to avoid letting Covet spend the user down to zero. The emergency floor should not be displayed on Home and should not be presented as a normal line item because it would make the interface feel like a budget worksheet. If the user taps into the Safe to Spend explanation, Covet may say something like “I’m keeping a small cushion untouched.” The exact emergency floor amount can appear only in a deeper explanation if the user asks for more detail. The default experience should be conclusion-first, not calculation-first.
Safe to Spend is calculated for the current pay cycle. The app should not frame the main number as a monthly budget unless the user’s income cadence is monthly. The pay cycle is the period from now until the next confirmed or expected income event. If the user is paid every other Friday, Safe to Spend applies until the next payday. If the user is paid weekly, it applies until the next weekly deposit. If the user has irregular income, the cycle should be derived from confirmed income and conservative cash runway rather than assuming a stable paycheck.
Daily pace is the simple user-facing pace underneath Safe to Spend. The user-facing calculation should be straightforward: Safe to Spend divided by the number of days until the next payday. If the user has $300 Safe to Spend and six days until payday, the visible pace is $50 per day. This keeps the interface understandable. Internally, however, Covet should use a smarter projected pace. The internal projected pace should account for uneven spending days, learned habits, and known upcoming events. If Covet knows the user usually spends $75 on Saturday brunch, it should not treat every day equally. It should quietly reserve for the weekend habit and monitor whether the user is spending too quickly before Saturday. The user does not need to see this full internal projection unless they ask why Covet is telling them to slow down.
The behavior buffer protects the user from their own likely behavior. It should be determined by primary archetype, selected strictness, and recent behavior. In v1, the primary archetype affects the engine first. The secondary archetype is shown to the user for identity and nuance, but the engine should not overcomplicate weighting at launch. Users can evolve over time as Covet observes their real behavior. A user who begins as The Spontaneous but consistently follows recommendations may gradually receive less restrictive behavior buffers. A user who begins as The Keeper but starts overspending after paydays may receive more protection. Archetype should guide the starting model; observed behavior should improve it.
Strictness should use elevated but clear names: Light, Balanced, and Protective. Light gives the user more freedom and a smaller behavior buffer. Balanced is the default. Protective gives Covet permission to reserve more aggressively, protect commitments earlier, and produce stronger purchase recommendations. Strictness should never change the truth of the user’s financial situation. It only changes how cautious Covet is when interpreting uncertainty and optional spending.
Commitment protection is the system that reserves money for what is coming up. Commitments include rent, essential bills, subscriptions, debt minimums, confirmed calendar events, travel, birthdays, school or work requirements, and user-entered future expenses. Commitments should be classified as hard, semi-hard, or soft. Hard commitments include rent, essential bills, insurance, required minimum debt payments, and non-negotiable obligations. Semi-hard commitments include confirmed events, travel, ID replacement, application fees, and expenses the user has explicitly marked as important. Soft commitments include lifestyle habits such as Pilates, brunch, beauty appointments, coffee routines, shopping patterns, and other repeat spending that matters to the user but can be reduced when money is tight.
Commitments should generally be protected gradually based on due date, amount, importance, and strictness. If the user selects Protective, Covet should reserve more aggressively and earlier. If the user selects Light, Covet can protect gradually while leaving more near-term flexibility. Hard commitments should always receive priority over soft commitments. Rent due in five days should be protected before brunch this weekend. A minimum credit card payment should be protected before a vault purchase. A confirmed birthday dinner may be protected before a flexible shopping desire, but can be adjusted if the user edits the expected amount.
Event protection should use estimates when an amount is missing, but every estimate must be editable. If a calendar event says “birthday dinner,” Covet can estimate a reasonable cost based on past dining, gifting, rideshare, and similar events, then present it for approval. If a calendar event says “concert,” Covet can consider ticket cost if detected, transportation, food, merch, and user history. If the user has exam week, Covet can anticipate higher convenience spending such as food delivery, coffee, rideshare, or wellness, but should not overprotect without evidence. The user should be able to approve, deny, or edit the estimate. Covet should learn from these edits.
Recurring habits should be handled differently from bills. A recurring Pilates class around $50 or brunch every weekend around $75 is part of the user’s real life, but it is not the same as rent. These habits should affect Safe to Spend as soft commitments. When the user is financially comfortable, Covet should quietly plan around them. When money is tight, Covet can reduce or pause soft commitments and notify the user. Example: “You’re a little short for the weekend. Let’s hold off on brunch this time, but you should be good again in two weeks.” The tone should be direct and protective, not shame-based.
Debt pressure should be based primarily on credit utilization and minimum payment protection. Covet must always protect required minimum debt payments before discretionary spending. Utilization should influence how strict the engine becomes because high utilization increases financial pressure and can indicate that the user is relying too heavily on credit. The initial utilization tiers should be: under 10% is healthy, 10% to 29% is normal, 30% to 49% is elevated, 50% to 74% is high, 75% to 89% is severe, and 90% or above is critical. These tiers should not be shown bluntly to the user as a score lecture. They should influence Safe to Spend, purchase decisions, and protective notifications.
The debt pressure adjustment should never create a positive Safe to Spend boost from credit availability. If utilization is healthy and the user consistently pays off the card, Covet may reduce the debt pressure penalty. If utilization is high, if minimum payments are coming due, or if card spending is accelerating, Covet should reduce Safe to Spend. If utilization is critical, Covet should become materially stricter and should discourage non-essential purchases that would worsen the situation. If a purchase would jeopardize a minimum payment or push the user into a dangerous utilization zone, Covet can say no, not now, or “Let’s not.”
Income cadence should be detected automatically from bank deposits and then confirmed by the user. Covet should identify likely payroll deposits, recurring income, gig income, transfers that are not true income, refunds, and one-off deposits. If Covet detects a pattern such as every other Friday, weekly, monthly, or irregular but recurring deposits, it should ask the user to confirm. The confirmation should be light: “Looks like payday is every other Friday. Confirm?” Once confirmed, payday cadence becomes a major input into Safe to Spend, daily pace, forecasting, and notifications.
Expected income should be treated carefully. Income that has not landed is not the same as cash. If payday is very near, such as Wednesday with a confirmed Friday payroll pattern, Covet can slightly relax the engine because the user is close to income. This should be implemented as a conservative pending income adjustment, not full cash inclusion. A reasonable v1 rule is that highly confident income expected within the next 72 hours may modestly improve status or Safe to Spend, but it should not be used to fund commitments due before the deposit lands. The adjustment should be capped and configurable. Covet should never let speculative income create unsafe spending permission.
Irregular income handling should use a conservative hybrid model. For freelancers, students, creators, interns, service workers, or users with inconsistent deposits, Covet should not assume the highest recent income month will repeat. It should use confirmed upcoming income when available, rolling historical deposits, and conservative averages. The engine should weigh recent reliable income more than one-off spikes. If income is inconsistent and no next deposit is confirmed, Covet should plan from current checking cash and known obligations first. It should communicate lower confidence rather than pretending the user has a stable pay cycle.
Covet should use obligation pressure rather than income amount alone to determine financial constraint. Covet should not classify users as financially constrained simply because income is low. Instead, it should calculate the share of income already committed to mandatory or semi-mandatory obligations. Mandatory obligations include rent, essential bills, minimum debt payments, required subscriptions, insurance, transportation required for work or school, and confirmed non-optional commitments. Optional habits like brunch, Pilates, beauty appointments, and shopping should not count as mandatory, though they can still affect Safe to Spend as soft commitments.
If 75% or more of confirmed income for the pay cycle is already accounted for by mandatory obligations, Covet enters a more protective mode. Internally this state should be called HIGH_OBLIGATION_PRESSURE. In this mode, Safe to Spend becomes stricter, optional recurring habits become easier to pause or reduce, and notifications become more protective. The user-facing language should never say “poor,” “low income,” or anything shame-based. Covet should simply behave more carefully. A user earning $500/month with only $40 in subscriptions and no debt should not be treated as high pressure because they may have meaningful discretionary room. A user earning $500/month with $400 in required bills and debt payments should be treated as high pressure because only $100 remains flexible. This rule allows Covet to work for high school students, college students, interns, freelancers, and early-career users without making simplistic income assumptions.
Pending transactions should reduce Safe to Spend immediately. Covet should not wait for transactions to post if the bank marks them as pending and the amount is reliable enough to affect available cash. If a pending transaction later disappears, changes amount, or posts differently, the engine should reconcile and update Safe to Spend. Large pending transactions should be treated as material until resolved. This prevents the user from receiving an overly generous Safe to Spend number after they have already spent money.
Transfers should not count as income or spending by default. Transfers between the user’s own accounts should be excluded from income detection and spending analysis. However, unclear transfers should be surfaced for review when they materially affect the engine. For example, if Covet cannot tell whether a $600 deposit is payroll, a refund, a transfer from savings, or a one-off gift, it should not automatically treat it as recurring income. It should ask the user to classify it if the answer matters.
Vaults should reduce Safe to Spend only when the user marks the vault as actively protected. A vault represents a desire or planned purchase, not automatically a mandatory obligation. If the user says a camera, jacket, trip, or laptop is actively being protected, the engine should reserve for it according to target amount, desired timing, and strictness. If the vault is only saved as a desire, it should not reduce Safe to Spend until the user activates protection or asks for an affordability plan.
The engine must map Safe to Spend and risk state into four primary statuses. YOU’RE GOOD means Safe to Spend is positive, hard commitments are protected, and the user has enough room for normal spending. TAKE IT EASY means the user is not in immediate danger but spending pace, upcoming obligations, debt pressure, or obligation pressure suggests caution. WAIT UNTIL PAYDAY means the user does not have enough current flexibility, but expected income will likely improve the situation soon. LET’S NOT… means the purchase or behavior would jeopardize bills, health, minimum debt payments, confirmed commitments, or essential stability. The system can also say direct versions such as “No, not now” or “You’re going to have to stop ordering delivery this week” when the user’s repetitive behavior is the issue. The tone should be honest and protective.
The engine explanation layer should reveal a simple breakdown, not the full sausage. When the user taps Safe to Spend, Covet can show a clean summary such as: “I’m protecting rent, your credit card minimum, brunch this weekend, and a small cushion. Your pace is $50/day until Friday.” The explanation should help the user trust the number without turning the product into a spreadsheet. Avoid full formulas, excessive category detail, and intimidating financial language in the default explanation.
Confidence scoring must exist internally and externally. Internally, the engine should maintain a percentage confidence score from 0 to 100 for Safe to Spend. External confidence should be simplified into High, Medium, and Still learning. High confidence means bank data is fresh, income cadence is confirmed, recurring obligations are well understood, and pending/ambiguous transactions are low. Medium confidence means enough data exists to produce a useful number, but some assumptions are unconfirmed. Still learning means bank data is new, income is unclear, calendar is disconnected, recurring patterns are unconfirmed, or transaction history is insufficient. Confidence should be visible in a subtle way when the user taps into Safe to Spend or reviews the explanation. It should not dominate Home.
Low confidence should make the engine more conservative. If Covet is unsure, it should not give the user an optimistic number to look polished. It should tighten buffers, request confirmation when needed, and explain that it is still learning. Confidence should improve as the user confirms patterns, connects calendar, accumulates transactions, confirms income cadence, and corrects wrong assumptions. Confidence should decline when bank connections break, data becomes stale, income changes unexpectedly, pending transactions are unusually high, or major calendar commitments are missing amounts.
The engine should recalculate after every transaction sync, bank balance update, confirmed commitment, denied commitment, edited recurring item, purchase check, calendar update, income cadence change, debt update, vault activation, and major pattern update. The engine should also recalculate after midnight or at the start of a new day because daily pace and time-to-due-date change even if no new transaction occurs. Recalculation should be efficient and should not require the UI to reload awkwardly. The user should feel that Covet is alive and current.
The engine should emit major change events for the Notification Engine. A major change should include any status change, any Safe to Spend change greater than 10%, or a dollar change that matters relative to the user’s current flexibility. In v1, use a lower dollar threshold when Safe to Spend is low. For example, if Safe to Spend is under $150, a $25 change may be material. If Safe to Spend is healthier, a $50 or larger change may be required unless the status changes. Major change detection should avoid noise. The Notification Engine will decide whether to notify, but the Financial Engine must expose the underlying event clearly.
When the engine must choose between competing priorities, it should protect in this order: rent and essential bills first, minimum debt payments second, confirmed non-optional commitments third, emergency floor fourth, recurring habits fifth, actively protected vaults sixth, and discretionary spending last. This priority order should be consistent across Safe to Spend, Purchase Check, Upcoming, Recurring, and notifications. If a discretionary purchase threatens anything above it, Covet should reduce approval confidence or say no.
The Financial Engine must support common real-life use cases from day one. If a user has a $50 Pilates habit most weeks, Covet should learn it as a recurring soft commitment and plan around it when possible. If the user spends around $75 on brunch most weekends, Covet should reserve for it internally but be willing to pause it when money is tight. If the user has a birthday, Covet should estimate and protect for likely gift, dinner, transportation, or buffer once confirmed. If the user needs a replacement ID or has a school/admin fee, Covet should treat it as a semi-hard commitment. If the user enters exam week, Covet should anticipate spending shifts such as coffee, delivery, rideshare, or wellness only when supported by user context or observed history. These examples are not edge cases. They are the product.
The required Financial Engine output should be a structured SafeToSpend object that the UI and Notification Engine can consume. It should include amount, payCycleStart, payCycleEnd, daysUntilNextIncome, dailyPace, internalProjectedPace, status, confidenceScore, externalConfidenceLabel, protectedHardCommitments, protectedSoftCommitments, debtPressureLevel, obligationPressureLevel, emergencyFloorApplied, behaviorBufferApplied, majorChangeFlags, explanationSummary, and lastCalculatedAt. The UI should not compute these independently. Home, Purchase Check, Upcoming, Activity, and Notifications should all read from the same engine output.
The acceptance criteria for this engine are strict. Safe to Spend must use checking cash only. It must reduce immediately for pending transactions. It must never increase because of credit card available balance. It must protect minimum debt payments before discretionary spending. It must detect and confirm income cadence. It must handle irregular income conservatively. It must use obligation pressure rather than income amount alone. It must produce external confidence labels and internal confidence scores. It must support hard, semi-hard, and soft commitments. It must expose a simple explanation without showing full calculations by default. It must recalculate after every meaningful financial, calendar, commitment, or purchase event. Most importantly, it must consistently answer the user’s real question: am I good?
