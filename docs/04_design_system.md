Covet’s design system must preserve the product’s core identity: calm, minimal, expensive, intelligent, and emotionally reassuring. The product should not look like a fintech dashboard. It should feel closer to Apple, Gentle Monster, Acne Studios, COS, and high-fashion editorial design than Rocket Money, Mint, Monarch, YNAB, or a generic AI finance app. The interface should communicate taste, restraint, and judgment. Every design decision should reduce cognitive load and support the user’s primary question: “Am I good?”
The canonical v1 visual direction is the latest white wallet Home screen direction provided by the founder. Claude/Codex should not reinterpret, redesign, modernize, gamify, or “improve” Covet into a standard fintech interface. The founder will provide UI/UX references, and those references should be treated as the source of truth. This specification defines the system behind that direction. If there is a conflict between generic mobile app convention and Covet’s visual direction, Covet’s direction wins unless accessibility, App Store requirements, or usability would be harmed.
Covet’s design language is luxury through restraint. The app should use generous whitespace, precise typography, subtle material effects, and minimal UI density. Empty space is not unused space. It is part of the emotional experience. The Home screen especially should remain sparse in v1. It should show the wordmark, small status language, Safe to Spend number, daily pace if enabled, wallet state, and one reassurance line. Later versions may add one small secondary item if truly useful, but Home must never become a widget dashboard.
The design system must explicitly ban charts, pie charts, category donuts, budget bars, multi-widget dashboards, net worth cards, spending category grids, and generic analytics modules from Home. These patterns can make users feel like they are responsible for interpreting their own finances. Covet’s job is to interpret first and present the answer. If deeper financial detail is needed, it should live behind progressive disclosure inside Activity, Upcoming, Safe to Spend explanation, or Settings.
Typography is one of Covet’s most important brand assets. The production app should use an elegant high-fashion serif for large money numbers, special display moments, and logo-adjacent brand elements. It should use a clean, modern sans-serif for labels, body text, navigation, settings, and utility information. The serif should feel editorial and refined, not decorative or old-fashioned. The sans-serif should feel precise and neutral, not tech-bro or playful. The Safe to Spend dollar amount should always be the largest text element on Home. Daily pace should be muted, secondary, and removable if the user disables pacing.
The COVET wordmark should appear on Home, Onboarding, and Settings. It does not need to dominate every secondary screen. Activity, Upcoming, and Purchase Check can use contextual headers because those screens are more functional. This keeps Home iconic while allowing utility screens to feel usable and mature. The logo should never be used as filler. If it appears, it should anchor the screen.
The glasses motif is a subtle brand asset. It can appear in onboarding, empty states, loading states, marketing surfaces, and occasional editorial moments, but it should not clutter core app screens. The glasses represent intelligence, observation, and “keeping an eye on your money.” They should never be overexplained in the interface. They should never become a cartoon mascot. They should feel like a quiet symbol users gradually associate with Covet.
The wallet is the main financial-state avatar across the app. It is not just decoration. It represents the user’s current financial condition. On Home, the wallet should be the central emotional object supporting the Safe to Spend number. In Purchase Check, Upcoming, and other screens, the wallet can appear more subtly when financial status is relevant. The wallet should not replace text clarity. It should reinforce the user’s status through material, glow, color, and motion.
Wallet states should map to status without using generic red/green finance language. The Good state can use the user’s selected primary color and matching wallet/aura treatment. The founder’s pink soft glow direction is one Good state, not the universal brand default. Take It Easy should use a pale amber or warm glow. Wait Until Payday should use muted silver or gray. Let’s Not should use a dark, low-glow, restrained red-black, or similarly serious treatment. Avoid bright green for good and aggressive red for bad. Covet should feel serious when needed, but never panicked.
Color should be quiet and material. Backgrounds should usually be soft white, warm off-white, deep black, or muted dark mode surfaces. Accent colors should feel like light, glass, metal, leather, or aura rather than flat SaaS colors. Status colors should be functional but restrained. Use contrast carefully so the app remains readable, but avoid visual alarm unless the user truly needs to act. Risk states should be serious, not frantic.
Dark mode is required for v1. Dark mode should not simply invert the light UI. It should feel like a luxury night mode: deep backgrounds, softened contrast, restrained glow, and careful typography. The wallet states should still read clearly in dark mode. Pink or user-selected primary states may glow more visibly in dark mode, but the effect should remain refined. Let’s Not in dark mode should be especially restrained; do not use harsh red panels or alarming full-screen warning states.
Spacing should be generous, especially on Home. Covet should show less information per screen if that makes the primary answer clearer. The user should never feel cramped. Secondary screens can be denser than Home, but still need editorial rhythm. Activity should feel like an elegant feed with sections, not a ledger. Upcoming should separate Upcoming Events, Recurring, and Vaults as distinct sections. Settings can be more conventional but should still feel calm and polished.
Buttons should not look like generic rounded SaaS CTAs. Covet should use two action systems. Lightweight decisions can use minimal text or icon actions, such as a check and X for confirming detected events or patterns. Larger decisions can use tactile luxury cards: soft rectangular surfaces, thin borders, generous padding, subtle pressed states, and precise labels. Think COS product card meets Apple settings cell. Do not use loud gradient buttons, chunky fintech pills, or playful gamified controls.
Confirming and denying detected events or patterns should use minimal check and X controls by default. These actions should be visually clear, fast, and low-friction. For higher-stakes actions, such as protecting a trip or editing a recurring bill, the interface can use labeled actions like “Protect,” “Edit,” or “Ignore,” but the default confirmation language should stay light. Covet should feel like it is asking for quick approval, not assigning homework.
Motion should feel like soft material movement. Use fades, slow glows, subtle wallet breathing, gentle slide transitions, and refined parallax where appropriate. Do not use bouncy startup-app animations, confetti, playful springs, excessive bounce, cartoon motion, or social-media-style reward loops. Motion should communicate confidence and presence. It should feel like opening a luxury object, not tapping through a game.
The wallet should have a subtle idle animation on Home. The animation should be slow enough that the user barely notices it at first: a soft breathing glow, tiny float, or material shimmer. It should not distract from the number. It should stop or reduce if the user enables reduced motion. On tap, the wallet can respond with a subtle press, glow shift, or expansion into a Safe to Spend explanation. Wallet motion should always feel calm and premium.
Loading states should be short, quiet, and specific. Preferred loading phrases include “updating your status…,” “reading what changed…,” “checking your week…,” “protecting what’s next…,” and “one moment….” The best default phrase is “updating your status…” because it ties directly to Covet’s Home language. Loading should not expose internal jobs like “syncing Plaid transaction batch.” If loading takes longer than expected, the app should explain the issue calmly.
Status language should remain low-key and small, similar to the current mockup. The status line can use the form “status : YOU’RE GOOD” or similar brand-consistent treatment. The status should be visible but not shout. The Safe to Spend number is the hero. The status is the judgment label. Together they should answer the user’s question immediately.
Navigation should remain Activity / Home / Upcoming, with Home centered. Activity contains Insights, Actions, and Transactions inside it. Do not change the bottom navigation to Insights / Home / Upcoming because insights are not always available, especially before the user has enough transaction history. Home should feel like the center of the product. Activity and Upcoming are supporting surfaces.
Activity should feel like an elegant feed with sections. The sections are Insights, Actions, and Transactions. Insights should only appear after sufficient data exists. Actions should contain items that need user attention, such as confirming a detected pattern or approving an event. Transactions should be clean, chronological, and reviewable, but not dominant. Activity should not become a notification log or accounting ledger.
Upcoming should visually separate Upcoming Events, Recurring, and Vaults. Upcoming Events includes calendar events, travel, birthdays, exams, and manually inputted events. Recurring includes bills, subscriptions, and recurring habits. Vaults includes desires or planned purchases. Each section should be scannable and calm. The interface should make it obvious what Covet is protecting, what needs confirmation, and what is optional.
Purchase Check should feel close to iMessage in ease, but Covet-branded in execution. It should use a simple bubble input, native keyboard behavior, clean message flow, and refined response cards. It should not feel like a cheap chatbot. Avoid AI avatars, excessive assistant personality, fake typing theatrics, or novelty chat UI. The user should be able to ask naturally, send a link, or upload a screenshot when supported. The answer should be visually clear: yes, wait, or no.
Empty states should be useful and calm, not cute. Good empty-state language includes “Nothing needs you right now,” “You’re covered for now,” or “Covet is still learning.” Empty states should reassure the user that the absence of content is intentional, not broken. Do not use quirky illustrations, forced humor, or overly cheerful copy.
Error states should be direct and clear. For technical issues, plain language is better than overly soft language. Example: “Connection failed. Please reconnect your bank.” If the issue affects Safe to Spend, say so clearly: “Your bank connection failed. Safe to Spend may be less accurate until you reconnect.” Errors should still look calm, but they should not hide the problem behind vague reassurance.
Financial risk visuals should be serious but never panic-red. For LET’S NOT or a bill-at-risk state, the interface can become darker, lower-glow, more muted, or more direct. It should not flash, shout, fill the screen with red, or use fear-based visuals. Covet should behave like a calm person telling the truth.
Onboarding should use the same aura, wallet, and glasses language as the rest of the product, combined with editorial quiz cards. The six-question archetype quiz should feel polished, not playful. Use progress indicators, but keep them minimal. The result screen should feel like a tasteful identity reveal, not a BuzzFeed quiz. The archetype copy should be flattering but honest.
Accessibility is required. Covet must support readable contrast, screen reader labels, tappable hit areas, Dynamic Type where feasible, reduced motion, and sufficient visible focus/touch states. Accessibility should not be treated as an aesthetic compromise. Done well, it will make Covet feel more polished. If a visual treatment is beautiful but unreadable or unusable, it does not belong in the production app.
Haptics should be subtle and rare. Use a soft haptic when confirming a commitment, tapping the wallet, or receiving a Purchase Check decision. A yes decision can have a light positive tap. A Let’s Not decision can have a slightly firmer but still restrained response. Do not overuse haptics. No sound should ship in v1. Sound adds asset complexity and annoyance risk, and haptics are enough for this release.
Iconography should be sparse and mostly line-based. Utility icons such as search, settings, check, X, lock, calendar, and refresh should be clean line icons. Custom editorial symbols can exist for brand moments, such as the glasses motif, wallet, or a refined lock/key motif, but not every icon should be custom. Too much custom iconography will make the app feel busy and less mature.
Design tokens should exist from the beginning. Use semantic tokens rather than hard-coded styling. Recommended token groups include color.background.primary, color.background.secondary, color.background.dark, color.text.primary, color.text.secondary, color.text.muted, color.status.good, color.status.takeItEasy, color.status.waitUntilPayday, color.status.letsNot, color.aura.primary, color.aura.warning, color.border.subtle, color.surface.card, spacing.xs, spacing.sm, spacing.md, spacing.lg, spacing.xl, spacing.screen, radius.card, radius.input, radius.pill, type.display.money, type.display.logo, type.body, type.label, motion.walletBreath, motion.fadeIn, motion.softSlide, motion.statusChange, and haptic.confirmation.
The design system should support user-selected primary color for Good-state wallet and aura treatments. This should not create chaos across the UI. The user-selected color can influence the wallet state, aura, selected accents, and subtle highlights. It should not recolor every button, every card, or every status. The brand must remain Covet even when the user customizes their primary color.
Claude/Codex must not redesign Covet. The implementation should follow the provided UI/UX and this design system exactly. Do not add generic fintech UI patterns. Do not add charts to Home. Do not add budget categories to Home. Do not add gamification, streaks, badges, loud gradients, playful mascots, dashboard widgets, or unnecessary screens. If a screen feels too empty, do not fill it automatically. Ask whether the emptiness is intentional. In Covet, restraint is a feature.
Part 4 acceptance criteria are strict. Home must remain sparse and status-first. The Safe to Spend amount must be the largest text element. Daily pace must be visually secondary and removable. Wallet states must map to financial status without bright green or aggressive red. The wallet must support subtle idle animation and reduced-motion fallback. Activity must be an elegant feed with Insights, Actions, and Transactions. Upcoming must separate Upcoming Events, Recurring, and Vaults. Purchase Check must feel like a refined iMessage-like interaction, not a cheap chatbot. Dark mode must exist. Accessibility must be respected. Error states must be direct. Empty states must be calm. Design tokens must be used. The product must feel like Covet, not a generic fintech app.
