The Notification Engine is one of Covet’s core product systems. Covet is notification-first, which means the product should manage money quietly in the background and interrupt only when something important changes or when the user needs to act. The app should not rely on daily engagement. A successful Covet user may open the app less often because they trust Covet to tell them when it matters. Notifications are not marketing, engagement bait, or reminders to use the product. They are part of the money management service.
The default notification posture must be quiet. Covet should send zero to two notifications per day maximum under normal circumstances. This cap can be exceeded only for serious situations such as a bill or debt minimum at risk, a confirmed important commitment becoming unsafe, a bank connection disconnecting, a security issue, or another event that materially affects trust in the current Safe to Spend number. Users may make notifications more active in Settings, but the default experience should prove that Covet respects attention. If Covet is unsure whether to notify, it should not notify.
The Notification Engine should operate from structured events emitted by the Financial Engine, Calendar Engine, Context Engine, Purchase Check, Vaults, and connection health systems. The Notification Engine should not independently calculate Safe to Spend or infer financial risk from raw transactions. It should receive candidate events, evaluate materiality, apply suppression and batching rules, respect quiet hours and privacy settings, and decide whether to send a notification. This separation is important because Safe to Spend must remain consistent across Home, Activity, Upcoming, Purchase Check, and notifications.
Covet notification severity should use internal levels that are clear but still aligned with the brand. The recommended severity system is Note, Nudge, Review, and Protect. A Note is low urgency and informational, such as a meaningful Safe to Spend increase after payday or a commitment becoming covered. A Nudge is a useful behavioral redirect, such as spending pressure rising or a soft habit needing to pause. A Review asks the user to confirm something that affects the engine, such as a recurring charge, calendar event, or unclear transfer. Protect is reserved for urgent or high-importance situations, such as rent, debt minimum, insurance, confirmed travel, or a critical commitment being at risk. These names are mostly internal; the user does not need to see severity labels unless a future notification settings UI benefits from them.
The primary trigger types are status changes, Safe to Spend increases, Safe to Spend decreases, commitment protection changes, commitment risk, purchase check follow-ups, vault affordability, sale or discount opportunities, recurring pattern confirmations, calendar event confirmations, repetitive behavior interventions, soft commitment pause recommendations, connection health issues, and security issues. Every trigger must pass materiality rules before becoming a notification, except connection/security issues and high-risk Protect events, which should generally notify immediately unless blocked by user settings or operating system constraints.
All material status changes should create notification candidates. If the user moves from YOU’RE GOOD to TAKE IT EASY, Covet should notify because the user’s decision environment has changed. If the user moves from TAKE IT EASY to YOU’RE GOOD, Covet should also notify, especially after payday, because the user should not have to open the app to learn they have room again. WAIT UNTIL PAYDAY and LET’S NOT… states should notify when they materially affect what the user can do or when caused by a purchase, bill, debt pressure, or commitment risk.
Safe to Spend increases should notify when they are meaningful, especially after income lands. The ideal payday notification is simple and useful: “you’re good. safe to spend is $416.” The goal is to remove the need for the user to check the app after payday. Safe to Spend decreases should notify when the change is material. The v1 materiality rule is any status change, any Safe to Spend change greater than 10%, any decrease of $25 or more when Safe to Spend is under $150, or any decrease of $50 or more when Safe to Spend is healthier. These thresholds should create notification candidates that still pass suppression, batching, quiet hours, and privacy rules, unless the event is Protect severity.
Daily pacing must be user-adjustable and may be turned off in Settings. Covet should not treat spending speed as inherently bad because many users prefer one larger meaningful purchase over several small purchases. A user spending $150 on shoes may be making a better decision than spending the same amount across small impulse purchases. The Notification Engine should care about impact, not moralize frequency. Spending pace notifications should trigger only when the projected impact suggests the user will run out before payday, jeopardize commitments, exceed their selected strictness tolerance, or materially reduce Safe to Spend. Amount and consequence matter more than the number of swipes.
Repetitive behavior notifications are allowed when the pattern is clear, specific, and material. Covet can explicitly call out repeated behavior if it is affecting the user’s money. For example, “not delivery again this week. it’s starting to crowd out friday.” Or: “you’re going to have to stop ordering rideshares until payday.” This type of message should be used carefully. It should feel like a trusted person being honest, not an app shaming the user. The engine should require enough evidence before using repetitive behavior language, such as repeated transactions in the same category or merchant pattern combined with declining Safe to Spend or commitment pressure.
Soft commitment pause notifications are part of the core experience. If the user has a learned brunch habit, Pilates habit, beauty appointment pattern, shopping pattern, or another soft recurring commitment, Covet should normally plan around it. When money is tight, Covet can recommend pausing or delaying that habit. Example: “let’s hold off on brunch this weekend. you should be good again in two weeks.” This is exactly where Covet feels like it manages money for the user without forcing them to budget.
Commitment protection notifications should be allowed when a commitment becomes fully protected, especially if the commitment is important, near-term, or previously uncertain. Example: “your concert next friday is covered.” This creates reassurance and teaches the user that Covet is actively protecting their life. Commitment-at-risk notifications should always notify for important commitments unless impossible due to user settings or system restrictions. Rent, essential bills, insurance, debt minimums, required transportation, confirmed travel, and important user-confirmed events should use Protect severity when at risk.
Calendar and event confirmation notifications should be used sparingly. If the user has recently opened the app, newly detected events can appear as Actions inside Activity or Upcoming without a push notification. If the user has not visited the app in a while and the event may affect Safe to Spend, Covet may notify. Example: “does saturday’s birthday dinner need money set aside?” These notifications should use Review severity and should offer quick approve/deny behavior when possible.
Pattern confirmation notifications are allowed in v1, especially when confirming the pattern would materially improve Safe to Spend. This includes recurring charges, payday cadence, bills, subscriptions, brunch habits, Pilates habits, unclear transfers, or repeated behaviors that affect forecasting. The notification should not feel like homework. Bad: “Confirm recurring transaction pattern.” Better: “looks like pilates is usually $50. plan around it?” If the pattern is minor and does not affect the current pay cycle, it should wait in-app.
Purchase Check follow-up is a required notification type. If the user asks whether they can buy something and Covet says wait, Covet should remember the item and notify when it becomes safe, unless the user dismisses or opts out. Example: “you’re good for the jacket now.” This creates a clear loop: ask, wait, return when safe. The follow-up should be tied to the specific item, amount, and reason the decision changed, such as payday landing, a commitment being covered, or a bill posting lower than expected.
Vault and desire affordability notifications are allowed when the user has saved or actively protected an item. Covet should notify when a saved desire becomes affordable, when the expected affordability date changes materially, or when the item would no longer fit safely. Example: “the camera fits now without touching rent.” These notifications should be user-toggleable because not every user wants shopping-related alerts. If a vault is passive and not actively protected, notifications should be less frequent and should not compete with core money protection.
Sale and discount alerts can exist in v1, but they must be tightly controlled and user-toggleable. These notifications are especially risky because they can become spammy or encourage spending. A sale notification should only be sent if the item is saved, the discount is meaningful, the user has opted into deal alerts, and the purchase is currently safe or nearly safe. Covet should not notify about random sales. Covet is not a coupon app. Sale alerts should support already-expressed desire, not create artificial urgency.
Timing rules matter because users may already receive Apple Pay, Capital One, Amex, bank, merchant, or receipt notifications immediately after spending. Covet should generally wait a few minutes before sending transaction-triggered notifications unless there is a serious risk. This avoids stacking alerts at the exact moment of purchase and gives the engine time to batch related changes. A delay of three to ten minutes is appropriate for most transaction-triggered candidate notifications. Protect-level events can send sooner when delay would harm the user.
Quiet hours should default to 9 PM to 9 AM local time. During quiet hours, Covet should suppress or delay non-urgent notifications. The app should avoid emotionally stressful money notifications late at night by default. A Safe to Spend decrease, soft commitment pause, sale alert, pattern confirmation, or general insight should wait until morning. Protect-level notifications can break quiet hours if the user needs to act, such as a bank disconnect that invalidates Safe to Spend, a bill due tomorrow at risk, a security event, or a payment issue. Users can edit quiet hours in Settings.
Batching is required. If several related events happen close together, Covet should combine them into one clear notification rather than sending multiple pings. Example: “safe to spend is $280 after groceries and your phone bill.” Batching should preserve usefulness. It should not create vague summaries that hide the action needed. If one event is Protect severity and another is only a Note, the Protect event should control the timing and message priority.
Suppression rules must prevent redundant notifications. If the user has opened the app recently and already seen the updated Home status, Covet should suppress the corresponding notification. If the user has dismissed an Action in-app, Covet should not send a push notification asking for the same review. If Covet has already warned the user about a specific behavior or soft commitment within the current relevant period, it should not repeat the same message unless the situation worsens materially. Since spending speed matters less than impact, repeated warnings should be based on amount, risk, and consequence, not merely how many purchases happened.
Notification privacy must default to discreet. Money is sensitive, and lock screen previews should not expose exact balances, debt risk, or embarrassing behavior by default. The default preview can say “covet has an update” or “your status changed.” Inside the app, the message can show the full detail. Settings should allow users to choose notification privacy level: Full detail, Discreet, or Hidden. Full detail can show messages like “safe to spend is $416.” Discreet should hide exact amounts and sensitive categories. Hidden should show only a generic Covet notification.
Covet notifications should not use emoji in v1. The tone should be direct, calm, and human. Lowercase is allowed where it matches the brand, but the message must remain readable and trustworthy. The psychology of tone should vary by context. For judgment and recommendations, “I” language can be useful because it feels like delegated judgment: “i’d wait until friday.” For status updates, neutral language is better because it feels confident and factual: “you’re good. safe to spend is $416.” For behavioral redirects, use direct but non-shaming language: “let’s not do delivery again this week.” Avoid corporate bank language, moralizing language, therapy-speak, excessive cheerfulness, and fake friendliness.
The Notification Engine should never notify for small purchases, normal subscriptions posting as expected, minor Safe to Spend changes, generic insights, non-material pattern detections, or anything that does not require action or materially improve reassurance. It should not send daily summaries by default. It should not send streaks, engagement prompts, “come back” messages, or generic financial tips. It should not notify just because there is a new transaction. It should not notify about a chart, category, or insight that is not specific enough to matter.
Disconnected bank or calendar notifications should be handled differently. If a bank disconnects, Covet should notify immediately because Safe to Spend may no longer be trustworthy. The message should be discreet on the lock screen and clear in-app. Example: “bank connection needs a refresh. safe to spend may be less accurate.” Calendar disconnects are less urgent unless they affect upcoming commitments, but the user should still be notified if Covet relies on that calendar for planning. The system should also lower confidence until the connection is restored.
Notification preferences must live in Settings from day one. Users should be able to adjust quiet hours, notification strictness, daily pacing notifications, sale and discount alerts, vault notifications, privacy level, and whether Covet can send review prompts for patterns or events. The settings should be simple and not overwhelming. Defaults should favor restraint. The user should feel that Covet is careful with attention.
The Notification Engine should track success metrics without becoming engagement-obsessed. The relevant metrics are open rate, action taken, reduced overspending after intervention, user-rated helpfulness, suppression effectiveness, and opt-out rate. A notification is successful when it helps the user understand a meaningful change, take an action, avoid a problem, or feel reassured. High volume is not success. If users ignore, mute, or disable Covet notifications, the engine has failed.
The required Notification object should include id, userId, candidateEventId, triggerType, severity, title, body, privacyBody, createdAt, scheduledFor, sentAt, quietHoursApplied, batchedEventIds, suppressedReason, statusBefore, statusAfter, safeToSpendBefore, safeToSpendAfter, relatedCommitmentId, relatedVaultId, relatedPatternId, relatedPurchaseCheckId, actionType, and userResponse. The UI should not construct notification meaning independently. All notification text should come from the Notification Engine or an approved copy generation layer constrained by this spec.
The Notification Engine acceptance criteria are strict. Covet must default to quiet. Normal notification frequency must stay between zero and two per day unless serious conditions apply. All material status changes must create notification candidates. Payday Safe to Spend increases must notify. Material Safe to Spend decreases must notify unless suppressed for a valid reason. Commitment-at-risk events must notify. Repetitive behavior and soft commitment pause notifications must be specific and material. Purchase Check follow-ups and vault affordability notifications must work. Sale alerts must be toggleable. Quiet hours must default to 9 PM to 9 AM. Lock screen previews must default to discreet. Notifications must be batched, suppressible, and never sent for generic engagement. If the engine is unsure whether to notify, it must stay silent.
